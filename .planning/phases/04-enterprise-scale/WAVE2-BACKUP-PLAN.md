## Ground Rules (inherited from Phase 4 Backup Plan, Wave 2 additions)

1. The Phase 3 test suite (244+ passing unit tests) is the hard gate — no Wave 2 code ships if existing tests regress.
2. Wave 2 runs in two **parallel** tracks. Track A and Track B share no source files and can proceed independently. Their only shared boundary is `ExecutionContext` and `ToolACL`, covered in Section 4.
3. Each task gets its own feature branch. The wave integration branch `phase-4/wave-2` is the landing target. No code merges directly to `phase-3/integration` or `master` until the full wave is verified.
4. Annotated tags before and after every wave: `phase4-pre-wave-2` (placed now) and `phase4-post-wave-2` (placed after verification).
5. Rollback on a shared branch means `git revert -m 1 <merge-commit>`. History is never rewritten on shared branches.
6. The existing `PersistentBudgetStore` (SQLite, sync) already exists at `src/orchestra/cost/persistent_budget.py`. T-4.5 migrates it to the DD-2 pessimistic-locking design. The rollback for T-4.5 reverts that file only.

### 1.4 Key Material Safety (T-4.6)

Wave 2 generates Ed25519 and X25519 keypairs for `AgentIdentity`. Key material must **never** be stored in:

- plaintext environment variables in `values.yaml` or any Helm chart committed to git
- Python source files (even as test fixtures)

If a key is accidentally committed, it must be treated as compromised: rotate immediately, rewrite the commit via `git filter-repo` on a private branch, force-push (coordinating with all active developers), and tag the old commit as `SECURITY-INCIDENT-[date]` for audit trail.

## Section 6: Data Safety

### 6.1 Budget Ledger Migration Safety (T-4.5)

The existing `PersistentBudgetStore` uses two tables: `budgets` and `transactions`. The DD-2 implementation adds a `budget_ledger` table (double-entry rows) and changes `budget_accounts` (the new name for `budgets`) to support hierarchy.

**Migration strategy:**

1. Schema migration is **lazy** — it runs on first `PersistentBudgetStore.initialize()` after the T-4.5 code is deployed.
2. The migration checks for the new column before adding it (idempotent):
   ```python
   # In _init_db() — safe to call multiple times
   cursor.execute("PRAGMA table_info(budget_accounts)")
   columns = {row[1] for row in cursor.fetchall()}
   if "parent_tenant_id" not in columns:
       conn.execute("ALTER TABLE budget_accounts ADD COLUMN parent_tenant_id TEXT")
   if "allocated_usd" not in columns:
       conn.execute("ALTER TABLE budget_accounts ADD COLUMN allocated_usd REAL DEFAULT 0.0")
   ```
3. The old `transactions` table is not dropped — it is kept as a read-only audit trail. New spending records go into `budget_ledger`.
4. Rollback SQL is pre-committed at `scripts/migrations/rollback_t4_5_budget.sql` (see Section 1.3).

**Data loss risk:** None, because:
- The migration is additive (adds columns and a new table; drops nothing).
- The old `transactions` table is preserved.
- The rollback SQL removes only the additions.

**Concurrent access:** `PersistentBudgetStore` uses `BEGIN IMMEDIATE` (SQLite WAL mode) for write transactions. The migration itself runs inside a transaction and is safe against concurrent reads. If two processes attempt `initialize()` simultaneously (unlikely in practice), the `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` construct (SQLite 3.37.0+) is idempotent.

### 6.2 Agent Identity Key Material (T-4.6)

**Key generation:** `AgentIdentity.create()` generates an ephemeral Ed25519 + X25519 keypair from `cryptography.hazmat.primitives.asymmetric.ed25519.Ed25519PrivateKey.generate()` and `X25519PrivateKey.generate()`. These are CSPRNG-backed by the OS.

**Key persistence rules (Wave 2 scope):**
- **Ephemeral agents (`did:peer:2`):** Keys are generated fresh on each `AgentIdentity.create()` call. They are held in memory only for the duration of the agent's session. No persistence required.
- **Long-lived agents (`did:web`):** Keys are loaded from `SecretProvider` (T-4.6 subtask). In development, `SecretProvider` reads from environment variables (`ORCHESTRA_AGENT_ED25519_KEY`, `ORCHESTRA_AGENT_X25519_KEY` as base64-encoded raw bytes). In production, `SecretProvider` reads from Vault (`hvac` backend) or AWS Secrets Manager.
- **Test fixtures:** All test keypairs are generated inline in `@pytest.fixture` scope and never written to disk or committed to git.

**Rollback-specific key safety:** If T-4.6 is rolled back after a `did:web` agent has published an Agent Card signed with its key, that Agent Card remains valid until its `expires_at` timestamp. There is no emergency revocation mechanism in Wave 2 (Phase 5 concern per DD-3). If an immediate revocation is needed on rollback, the Agent Card's HTTP endpoint (`did.json`) must be taken offline manually.

**Accident response for committed keys:**
1. Rotate immediately: generate new keypair, deploy new `did.json`.
2. Invalidate the committed secret: revoke in Vault or rotate in AWS Secrets Manager.
3. Remove from git history: `git filter-repo --path <file-with-key> --invert-paths`.
4. Force-push (coordinate with team): `git push --force-with-lease origin <branch>`.
5. Tag the corrupted commit: `git tag SECURITY-INCIDENT-<date> <bad-commit-hash>` (for audit).

### 6.3 UCAN Token Expiry on Rollback (T-4.7)

UCAN tokens issued by T-4.7 are short-lived JWTs (1–60 minutes, default 60 min). If T-4.7 is rolled back mid-operation, any in-flight UCAN tokens will simply expire. There is no persisted state to clean up:

- UCAN tokens are passed in-band (inside `ExecutionContext.ucan_token`).
- They are not stored in any database.
- The `ucan_call_counters` dict on `ExecutionContext` is ephemeral (per-run, in-memory only per DD-4).

**Result:** T-4.7 rollback is clean with no data artifacts.

### 7.4 Budget DB Corruption (T-4.5 Production Edge Case)

**Scenario:** The pessimistic lock in `PersistentBudgetStore` is acquired, the server crashes mid-transaction (after deducting balance, before writing the `budget_ledger` row). On restart, a tenant's balance is reduced but no ledger row exists, creating a ghost deduction.

**Detection:** The reconciliation query `SELECT SUM(amount_usd) FROM budget_ledger WHERE tenant_id = ?` does not match `budget_accounts.balance_usd`. Expose this as a `/health/budget-reconcile` endpoint that returns mismatches.

**Recovery without code rollback:**
```sql
-- Manual reconciliation: recompute balance from ledger rows
UPDATE budget_accounts
SET balance_usd = (
    SELECT COALESCE(SUM(amount_usd), 0.0)
    FROM budget_ledger
    WHERE budget_ledger.tenant_id = budget_accounts.tenant_id
      AND budget_ledger.entry_type = 'debit'
);
```

**Prevention (defense in depth):** T-4.5 must use a SQLite `SAVEPOINT` or a two-phase write:
1. Insert `budget_ledger` row first (with `status='pending'`).
2. Deduct from `budget_accounts`.
3. Update `budget_ledger` row `status='committed'`.
4. On startup, find `status='pending'` rows and roll them back (compensating transaction).

This is the "saga" pattern for single-process SQLite and avoids the ghost deduction scenario.

### 7.5 Incomplete Identity Propagation (T-4.6/T-4.7 Partial Failure)

**Scenario:** T-4.6 is merged and working. T-4.7 UCAN delegation is merged but the `ToolACL.is_authorized(ucan=...)` path has a bug that causes all tool calls to return `False` (deny-all), breaking existing workflows.

**Detection:** Existing `tests/unit/test_acl.py` must cover the `ucan=None` default path explicitly. If that test passes, the bug is in the new UCAN path only, not the backward-compatible path.

**Immediate mitigation (without code rollback):** Add a feature flag to `ExecutionContext.config`:

```python
# In compiled.py, when calling ToolACL.is_authorized():
use_ucan = ctx.config.get("enable_ucan_enforcement", False)
ucan_token = ctx.ucan_token if use_ucan else None
acl.is_authorized(tool_name, ucan=ucan_token)
```

Set `enable_ucan_enforcement=False` in all existing workflow configs. This restores the pre-T-4.7 behavior without reverting code, buying time to fix the bug.

**Code rollback if the flag mitigation is insufficient:**
```bash
git revert -m 1 <t-4.7-merge-commit>
```
T-4.6 remains merged and intact. Only `ucan.py`, `delegation.py`, and the `acl.py` `ucan` parameter are reverted.
